Session Summary — July 10, 2026
Started: late night
=================================

- Added opening fields directly to the Game model (no separate Opening model)
  - eco_code: CharField for ECO code (e.g. "B90")
  - opening_line: CharField for full opening name (e.g. "Sicilian Defense: Najdorf Variation")
  - opening_family: CharField derived by splitting opening_line on ":" (e.g. "Sicilian Defense")

- Added ECO lookup to pgn_parser.py
  - Loads all 5 ECO JSON files into one dictionary at module level (read once on startup)
  - Inside the move loop, after board.push(move), checks board.fen() against eco_lookup
  - Keeps updating to the most specific match — last match wins
  - After the move loop, saves eco_code, opening_line, and opening_family to the game

- Explored alternative approaches for opening family
  - Considered using the first ECO code match as the family name
  - Found that ECO codes can span multiple unrelated openings (e.g. A00 covers Amar, Amsterdam, etc.)
  - Went with colon-split approach instead — more reliable since naming convention is consistent

- Confirmed python-chess is the right library for this
  - Only library that tracks board state and generates FENs
  - pgnparse and pgn2data don't support move-by-move board tracking

- Added FEN board diagrams to DEVLOG.md
  - ASCII board showing the Amar Gambit FEN with rank-by-rank breakdown
  - Three-board sequence showing 1. e4 e5 with FEN lookup at each step

- Ran migration 0007 to add the three new fields to the database

- Deleted old games from the database and re-uploaded with opening data

- Tried first-match approach for opening family — too generic
  - Any 1. e4 e5 game got "King's Pawn Game" regardless of actual opening
  - Reverted to colon-split on the last match (e.g. "Italian Game: Two Knights" → "Italian Game")
